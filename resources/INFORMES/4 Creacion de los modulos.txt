2.1. Diseño conceptual del módulo
------------------------------------------------------------
Primero analizamos qué debía representar realmente una pantalla de Gestión de Estudiantes dentro de una Unidad Educativa Técnico Humanístico.

Se definió que este módulo no debía verse como una tabla simple, sino como un espacio donde distintos roles pudieran interpretar la información del estudiante:

Docente:
- Identificar rápidamente al estudiante.
- Ver curso, paralelo, especialidad, estado e inscripción.
- Acceder al historial académico.

Director:
- Ver población estudiantil.
- Controlar matrícula, inscripción y estado institucional.

Autoridad educativa:
- Ver trazabilidad.
- Revisar estados académicos y administrativos.
- Confirmar que no exista eliminación física de información sensible.

Con esto se definió que la interfaz debía tener un enfoque más serio, académico e institucional.

La conclusión fue clara:

El nombre del permiso debe ser exactamente igual en:
- Seeder de permisos
- Middleware de ruta
- Directiva @can del menú

2.6. Corrección conceptual de curso y paralelo
------------------------------------------------------------

Después analizamos un problema importante del modelo académico:

El usuario indicó que no podía relacionar directamente curso y paralelo de manera simple, porque esa organización ya estaba relacionada con:

plan_asignatura
plan_especialidad
gestión académica

Se entendió que, si curso y paralelo se modelaban mal, podían aparecer duplicaciones como:

6 veces A
6 veces B

Por eso se aclaró que:

Curso y paralelo deben funcionar como catálogos base.
La combinación curso + paralelo + turno + gestión debe manejarse desde planes o inscripción.

Esto ayudó a evitar redundancia y problemas de normalización.

2.7. Corrección del seeder de paralelo
------------------------------------------------------------    

Se revisó el ParaleloSeeder.

El problema era que estaba generando paralelos por cada curso:

foreach ($cursos as $codCur) {
    foreach ($paralelos as $paralelo) {
        ...
    }
}

Pero la tabla paralelo no tenía cod_cur, por lo tanto estaba creando muchos registros repetidos:

A, B, C, D
A, B, C, D
A, B, C, D
...

La conclusión fue:

La tabla paralelo debe tener solo los paralelos base:
A
B
C
D

No debe duplicarse por curso.

------------------------------------------------------------

3.1. Análisis conceptual de la bitácora
------------------------------------------------------------

Luego pasamos al módulo Bitácora Institucional.

Primero analizamos qué debía representar este campo desde una mirada educativa, administrativa y de auditoría.

Se concluyó que la bitácora debía servir para:

Registrar acciones del sistema.

5.8. Lógica correcta de cierre académico

------------------------------------------------------------

Se definió que para cerrar una gestión:

La gestión debe estar:

ACTIVO

Y no debe tener:

Inscripciones pendientes
Planes de asignatura incompletos
Planes de especialidad incompletos

Se consideró incompleto un plan de asignatura cuando:

est_pas != ACTIVO
hor_pas IS NULL
hor_pas <= 0

Se consideró incompleto un plan de especialidad cuando:

est_pes != ACTIVO
hor_pes <= 0

Luego se implementó el cierre real:

'est_gea' => 'CERRADO'

Esto reemplazó la lógica anterior que solo mostraba un mensaje informativo.

5.9. Corrección del error del seeder eliminado

------------------------------------------------------------

El usuario ejecutó:

sail artisan migrate:fresh --seed

Las migraciones corrieron correctamente, pero falló el seeder:

include(.../GestionAcademicaSeeder.php): Failed to open stream: No such file or directory

La causa fue que el archivo GestionAcademicaSeeder.php había sido eliminado, pero todavía estaba llamado desde:

database/seeders/DatabaseSeeder.php

La solución fue eliminar la línea:

GestionAcademicaSeeder::class,

del DatabaseSeeder.php.

Se reforzó la decisión:

Gestión Académica debe crearse desde el sistema, no desde seeder.

Porque si se crea desde seeder puede generar:

Años duplicados
Gestión activa falsa
Estados iniciales incorrectos
Confusión en producción

Decisión 3: La bitácora debe ser interpretada

------------------------------------------------------------

La bitácora no debe mostrar solo acciones técnicas.

Debe transformar:

CREAR_GESTION_ACADEMICA

en:

Se registró una nueva gestión académica.

------------------------------------------------------------

Decisión 4: Periodo de evaluación es catálogo general

------------------------------------------------------------

Como periodo_evaluacion no tiene cod_gea, no se debe tratar como dependiente de gestión.

Por ahora:

periodo_evaluacion = catálogo institucional general

------------------------------------------------------------

Decisión 5: Gestión Académica se crea desde el sistema

------------------------------------------------------------

Se eliminó el seeder de gestión académica.

La razón:

Evitar gestiones falsas.
Evitar años duplicados.
Evitar estado ACTIVO insertado sin control.
Forzar creación mediante validaciones Livewire.

------------------------------------------------------------

Decisión 6: Cierre académico responsable

------------------------------------------------------------

Una gestión no se cierra directamente.

Primero se revisa:

Inscripciones pendientes
Planes de asignatura incompletos
Planes de especialidad incompletos

Solo si todo está en 0, se permite cerrar.

-------------------------------11. Reflexión del desarrollador----------------------------------

Esta etapa fue una de las más importantes del proyecto SAVP: TIS-3, porque se pasó de tener una base de datos estructurada a comenzar a construir módulos administrativos reales.

El aprendizaje principal fue entender que un sistema educativo no puede reducirse a CRUDs simples. Cada módulo debe representar una responsabilidad institucional.

La Gestión de Estudiantes no es solo registrar estudiantes; es controlar matrícula, inscripción, estado académico y trayectoria.

La Bitácora no es solo guardar logs; es garantizar trazabilidad, seguridad y responsabilidad institucional.

La Gestión Académica no es solo guardar un año; es controlar todo un ciclo educativo, sus inscripciones, planes, estructura, cierre e historial.

También se aprendió algo muy importante: no se deben inventar datos para que una interfaz se vea bonita. Un sistema serio debe mostrar la verdad de su base de datos, aunque al principio esté vacía. Si no hay registros, debe decirlo. Si no hay actividad, debe mostrarlo. Si no existe una relación, no se debe fingir que existe.

Este avance demuestra que el sistema está creciendo con criterio. Ya no se trata solo de programar pantallas, sino de construir una herramienta que pueda ser defendida académicamente, entendida por docentes, usada por administrativos y revisada por una autoridad educativa.

Mi versión personal

Hoy avanzamos bastante en el proyecto SAVP: TIS-3. No solo hicimos código, sino que entendimos mejor cómo debe funcionar un sistema educativo real.

Aprendí que cada módulo tiene que tener sentido dentro de la institución. No es suficiente crear una tabla y mostrarla en pantalla. Hay que pensar quién la usará, qué necesita ver, qué errores se deben evitar y cómo se va a mantener la trazabilidad.

También aprendí que no debo mostrar datos falsos solo para que la interfaz se vea llena. Si el sistema no tiene información, debe decirlo claramente. Eso hace que el sistema sea más serio y más confiable.

En Gestión Académica entendí que un año escolar no es solo un número. Es un ciclo completo donde se organizan inscripciones, cursos, paralelos, planes, especialidades y cierre institucional. También entendí que cerrar una gestión no debe ser algo simple, porque representa cerrar un periodo importante de la institución.

En Bitácora aprendí que registrar acciones no sirve si nadie las entiende. Por eso es importante humanizar los eventos y mostrar lo que pasó de forma clara.

Siento que este proyecto cada vez se vuelve más grande, pero también más real. Al inicio parecía que solo estaba haciendo pantallas, pero ahora veo que estoy construyendo una estructura que puede ayudar a organizar mejor una institución educativa.

Todavía falta mucho, pero esta etapa me demuestra que el sistema ya tiene una base fuerte. Cada error corregido, cada migración ajustada y cada vista mejorada me ayuda a entender mejor cómo se construye software de verdad.

Hoy no solo avancé en código. Avancé como desarrollador.
